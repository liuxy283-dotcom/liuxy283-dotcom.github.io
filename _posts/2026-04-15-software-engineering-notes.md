## 软件架构分层

一个典型的软件项目通常采用三层架构：

### 1. 表现层（Presentation Layer）
- 负责用户界面展示
- 处理用户输入和输出
- 与业务逻辑层通信

### 2. 业务逻辑层（Business Logic Layer）
- 实现核心业务规则
- 协调数据访问
- 封装领域逻辑

### 3. 数据访问层（Data Access Layer）
- 负责数据库操作
- 提供数据持久化
- 实现 ORM 映射

## 类关系类型

| 关系类型 | 符号 | 含义 |
|---------|------|------|
| 继承（泛化） | 空心三角箭头 | is-a 关系 |
| 实现 | 空心三角虚线箭头 | 实现接口 |
| 关联 | 实线箭头 | 引用关系 |
| 聚合 | 空心菱形 + 实线 | 整体-部分，可独立存在 |
| 组合 | 实心菱形 + 实线 | 整体-部分，生命周期相同 |
| 依赖 | 虚线箭头 | 使用关系 |

## UML 类图绘制要点

1. **类名**：首字母大写的名词
2. **属性**：可见性 + 名称 + 类型
3. **方法**：可见性 + 名称 + 参数 + 返回类型
4. **关系方向**：箭头指向被依赖/使用的类

## 案例分析：小米便签项目

在分析小米便签开源项目时，发现其核心类关系：

- `NotesListActivity` → 继承 `AppCompatActivity`
- `NotesListAdapter` → 组合 `List<Note>`
- `NoteProvider` → 继承 `ContentProvider`
- `WorkingNote` → 依赖 `Note`

## 总结

掌握分层架构和类关系分析是软件工程的基础能力。在绘制类图时，关键是准确把握类之间的语义关系，而不是机械地使用符号。
